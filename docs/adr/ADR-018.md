# ADR-018 - Acesso Público Autenticado (NextAuth + Google OAuth)

**Status:** Rejeitada na implementação (2026-07-15) — ver nota de implementação. Revertida para o modelo de acesso da ADR-016 (proxy CLI + IAM). Débito técnico registrado para uma abordagem futura de autenticação usuário/senha controlada pela própria aplicação.

## Contexto

O acesso à `editorial-ui` hoje (ADR-016) depende de `gcloud run services proxy` rodando num terminal — o serviço é `--no-allow-unauthenticated`, protegido por `roles/run.invoker` restrito a `contato@banhoseterapias.com`. Isso funciona, mas exige abrir um terminal e manter um processo rodando toda vez que se quer acessar a plataforma, o que já se mostrou uma fonte de atrito e de bugs (ver nota de implementação da ADR-016: o próprio proxy introduziu um bug de CSRF em Server Actions por reescrever `x-forwarded-host`).

O usuário quer um acesso hospedado, com URL fixa, aberto direto no navegador, protegido por login — sem depender de CLI. Entre as opções levantadas (GCP Identity-Aware Proxy, autenticação embutida na própria aplicação, manter o proxy CLI), o usuário escolheu explicitamente **autenticação embutida na aplicação via NextAuth + Google OAuth**, por não exigir infraestrutura nova nem custo adicional de GCP (IAP normalmente exige um Load Balancer HTTPS externo na frente do Cloud Run, com custo mensal).

## Decisão

1. **NextAuth (Auth.js), provider Google, sessão JWT.** Sem adapter de banco — não existe tabela de usuários no schema `editorial.*`, e criar uma só para autenticar 2 e-mails conhecidos seria complexidade desnecessária. A sessão vive inteiramente no JWT assinado, sem estado no Postgres.

2. **Allowlist fixa de e-mails**, lida de uma env var (`ALLOWED_EMAILS`, lista separada por vírgula): `thiago.6286@gmail.com`, `contato@banhoseterapias.com`. O callback `signIn` do NextAuth rejeita qualquer login fora dessa lista — controle de acesso continua existindo, só que dentro da aplicação em vez de via IAM.

3. **`middleware.ts` protege todas as rotas**, incluindo as Route Handlers em `app/api/*` (`/api/concepts`, `/api/segments`, `/api/knowledge-cards`, `/api/search`) — não só as páginas. Sem isso, essas rotas ficariam publicamente acessíveis assim que o Cloud Run for `--allow-unauthenticated`.

4. **Cloud Run passa a ser público** (`--allow-unauthenticated`). O `roles/run.invoker` restrito hoje deixa de ser o mecanismo de controle de acesso — é removido. Todo o controle de quem entra passa a ser feito pela aplicação (passo 2).

5. **Secrets novos no Secret Manager**, seguindo o padrão já usado por `editorial-api-key`: `nextauth-secret` (chave de assinatura do JWT, gerada aleatoriamente) e `google-oauth-client-secret` (do OAuth 2.0 Client ID criado no GCP, tipo "Web application", com redirect URI `https://<url-do-serviço>/api/auth/callback/google`). Acesso restrito à service account `editorial-ui@thiago-ai-platform.iam.gserviceaccount.com` via `secretmanager.secretAccessor`, mesmo padrão já usado.

6. **Correção obrigatória de regressão**: `editorial-ui/next.config.ts` hoje trava `experimental.serverActions.allowedOrigins: ["localhost:8081"]` — um workaround específico do proxy CLI (ver nota de implementação da ADR-016). Com acesso público direto, isso precisa virar o hostname real do Cloud Run (`editorial-ui-592824114603.southamerica-east1.run.app` ou domínio customizado, se houver). Sem essa correção, toda Server Action (criar projeto, salvar capítulo, aprovar, revisar) volta a quebrar com 500.

## Consequências

- A plataforma passa a ter uma URL fixa acessível direto no navegador, com login Google — resolve o atrito do proxy CLI.
- O controle de acesso deixa de ser imposto pela infraestrutura (IAM do Cloud Run) e passa a ser responsabilidade do código da aplicação — isso é uma superfície nova a manter (allowlist, secrets de OAuth, sessão), mas evita custo de infraestrutura adicional (Load Balancer para IAP).
- Reforça a necessidade de rigor: um bug na allowlist ou no middleware expõe o acervo publicamente, diferente de hoje onde o IAM barra na borda antes mesmo de a aplicação rodar. Testes explícitos de "usuário fora da allowlist é rejeitado" e "rota de API sem sessão retorna 401/redirect" são obrigatórios antes de considerar esta ADR implementada.
- Esta ADR é pré-requisito prático para ADR-019, ADR-020 e ADR-021: todas se beneficiam de poder ser testadas via URL pública, sem depender do proxy CLI.

## Nota de implementação (2026-07-15) — tentada, revertida

O desenho acima foi implementado por completo (NextAuth v5/Auth.js, provider Google, `proxy.ts` — nesta versão do Next.js `middleware.ts` foi renomeado para `proxy.ts`, exportando `proxy` em vez de `middleware`, protegendo páginas e `app/api/*`, allowlist via `ALLOWED_EMAILS`, OAuth 2.0 Client ID "plataforma-editorial-filosofica" criado no GCP, secrets `nextauth-secret`/`google-oauth-client-secret`) e depois **revertido** por dois problemas descobertos só em produção, um deles estrutural e não contornável em código:

1. **Bloqueio real, não previsto no desenho original:** o projeto GCP tem uma política de organização (Domain Restricted Sharing, ligada ao Cloud Identity/Workspace de `banhoseterapias.com`) que impede conceder **qualquer** binding IAM a identidades fora da organização — inclusive `allUsers` (tornar o Cloud Run público, passo 4 do desenho original) e inclusive a conta pessoal `thiago.6286@gmail.com` especificamente (testado e confirmado: `FAILED_PRECONDITION: One or more users named in the policy do not belong to a permitted customer`). Confirmado também que essa conta pessoal nunca teve papel IAM neste projeto — todo acesso via `gcloud`/proxy sempre operou como `contato@banhoseterapias.com`, independente da conta logada no navegador.
2. **Conflito estrutural entre IAM do Cloud Run e callback OAuth**, descoberto ao tentar restringir o acesso só a `contato@banhoseterapias.com` (mantendo `--no-allow-unauthenticated`): o passo final do fluxo OAuth é o **servidor do Google** redirecionando o navegador direto para `https://<host-real>/api/auth/callback/google` — essa navegação do navegador não passa pelo túnel do `gcloud run services proxy` (que só intermedia o tráfego do desenvolvedor, não o redirecionamento feito pelos servidores do Google), então bate direto no IAM do Cloud Run e recebe 403, mesmo sendo a conta autorizada. `roles/run.invoker` foi desenhado para acesso máquina-a-máquina com token, não para fluxo de login interativo redirect-based no navegador — a ferramenta correta para isso é o Identity-Aware Proxy (IAP), que resolve a sessão na borda via cookie antes de chegar no Cloud Run.

**Decisão de reversão:** removido `auth.ts`, `proxy.ts`, o Route Handler `app/api/auth/[...nextauth]`, a dependência `next-auth` e as env vars/secrets relacionadas do deploy. `editorial-ui` voltou a rodar exatamente como na ADR-016 (`--no-allow-unauthenticated`, IAM restrito a `contato@banhoseterapias.com`, acesso via `gcloud run services proxy --port=8081`). Os secrets `nextauth-secret`/`google-oauth-client-secret` foram apagados do Secret Manager. O OAuth 2.0 Client ID "plataforma-editorial-filosofica" e a tela de consentimento OAuth criados no GCP Console **não foram apagados** (inertes, sem custo, mas também sem uso agora — ver [docs/GCP_INFRAESTRUTURA.md](../GCP_INFRAESTRUTURA.md)).

**Débito técnico registrado (decisão explícita do usuário):** a próxima tentativa de acesso hospedado sem CLI deve ser **autenticação usuário/senha controlada pela própria aplicação** (credenciais próprias, sessão via cookie, sem redirect para um provedor OAuth externo) — esse desenho não tem o problema #2 acima, porque toda a troca de credenciais acontece dentro da mesma origem que o navegador já está acessando, sem um servidor terceiro redirecionando o navegador para fora do túnel. Ainda esbarraria no problema #1 se a meta continuar sendo tornar o Cloud Run público (`allUsers`) — path viável só se o Cloud Run permanecer IAM-gated para `contato@banhoseterapias.com` (que é quem hoje efetivamente opera este projeto GCP) e a autenticação de aplicação for uma camada adicional sobre isso, ou se o IAP for reconsiderado especificamente para essa conta (a política de organização não deveria bloquear IAP para um membro real da organização, diferente do que ocorreu com `allUsers`/conta pessoal).
