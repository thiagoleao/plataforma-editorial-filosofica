# ADR-018 - Acesso Público Autenticado (NextAuth + Google OAuth)

**Status:** Proposta (2026-07-15)

## Contexto

O acesso à `editorial-ui` hoje (ADR-016) depende de `gcloud run services proxy` rodando num terminal — o serviço é `--no-allow-unauthenticated`, protegido por `roles/run.invoker` restrito a `contato@banhoseterapias.com`. Isso funciona, mas exige abrir um terminal e manter um processo rodando toda vez que se quer acessar a plataforma, o que já se mostrou uma fonte de atrito e de bugs (ver nota de implementação da ADR-016: o próprio proxy introduziu um bug de CSRF em Server Actions por reescrever `x-forwarded-host`).

O usuário quer um acesso hospedado, com URL fixa, aberto direto no navegador, protegido por login — sem depender de CLI. Entre as opções levantadas (GCP Identity-Aware Proxy, autenticação embutida na própria aplicação, manter o proxy CLI), o usuário escolheu explicitamente **autenticação embutida na aplicação via NextAuth + Google OAuth**, por não exigir infraestrutura nova nem custo adicional de GCP (IAP normalmente exige um Load Balancer HTTPS externo na frente do Cloud Run, com custo mensal).

## Decisão

1. **NextAuth (Auth.js), provider Google, sessão JWT.** Sem adapter de banco — não existe tabela de usuários no schema `editorial.*`, e criar uma só para autenticar 2 e-mails conhecidos seria complexidade desnecessária. A sessão vive inteiramente no JWT assinado, sem estado no Postgres.

2. **Allowlist fixa de e-mails**, lida de uma env var (`ALLOWED_EMAILS`, lista separada por vírgula): `thiago.6286@gmail.com`, `contato@banhoseterapias.com`. O callback `signIn` do NextAuth rejeita qualquer login fora dessa lista — controle de acesso continua existindo, só que dentro da aplicação em vez de via IAM.

3. **`middleware.ts` protege todas as rotas**, incluindo as Route Handlers em `app/api/*` (`/api/concepts`, `/api/segments`, `/api/knowledge-cards`, `/api/search`) — não só as páginas. Sem isso, essas rotas ficariam publicamente acessíveis assim que o Cloud Run for `--allow-unauthenticated`.

4. **Cloud Run passa a ser público** (`--allow-unauthenticated`). O `roles/run.invoker` restrito hoje deixa de ser o mecanismo de controle de acesso — é removido. Todo o controle de quem entra passa a ser feito pela aplicação (passo 2).

5. **Secrets novos no Secret Manager**, seguindo o padrão já usado por `editorial-api-key`: `nextauth-secret` (chave de assinatura do JWT, gerada aleatoriamente) e `google-oauth-client-secret` (do OAuth 2.0 Client ID criado no GCP, tipo "Web application", com redirect URI `https://<url-do-serviço>/api/auth/callback/google`). Acesso restrito à service account `editorial-ui@thiago-ai-platform.iam.gserviceaccount.com` via `secretmanager.secretAccessor`, mesmo padrão já usado.

6. **Correção obrigatória de regressão**: `editorial-ui/next.config.ts` hoje trava `experimental.serverActions.allowedOrigins: ["localhost:8081"]` — um workaround específico do proxy CLI (ver nota de implementação da ADR-016). Com acesso público direto, isso precisa virar o hostname real do Cloud Run (`editorial-ui-592824114603.southamerica-east1.run.app` ou domínio customizado, se houver). Sem essa correção, toda Server Action (criar projeto, salvar capítulo, aprovar, revisar) volta a quebrar com 500.

## Consequências

- A plataforma passa a ter uma URL fixa acessível direto no navegador, com login Google — resolve o atrito do proxy CLI.
- O controle de acesso deixa de ser imposto pela infraestrutura (IAM do Cloud Run) e passa a ser responsabilidade do código da aplicação — isso é uma superfície nova a manter (allowlist, secrets de OAuth, sessão), mas evita custo de infraestrutura adicional (Load Balancer para IAP).
- Reforça a necessidade de rigor: um bug na allowlist ou no middleware expõe o acervo publicamente, diferente de hoje onde o IAM barra na borda antes mesmo de a aplicação rodar. Testes explícitos de "usuário fora da allowlist é rejeitado" e "rota de API sem sessão retorna 401/redirect" são obrigatórios antes de considerar esta ADR implementada.
- Esta ADR é pré-requisito prático para ADR-019, ADR-020 e ADR-021: todas se beneficiam de poder ser testadas via URL pública, sem depender do proxy CLI.
