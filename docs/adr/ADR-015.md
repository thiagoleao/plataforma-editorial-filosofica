# ADR-015 - Publicação Final da Obra

**Status:** Implementada (2026-07-20 — revisão do desenho original de 2026-07-14, escrito antes da ADR-020 existir)

## Contexto

O desenho original desta ADR propunha DOCX como formato primário de **edição**: gerar um documento Word a partir da montagem do capítulo para que o curador desse o polimento final ali, antes de qualquer distribuição. Isso ficou obsoleto — a ADR-020 já implementou um editor de manuscrito dentro da própria plataforma (`chapters.manuscript_content`, Tiptap), com blocos de canalização travados e texto de ligação livremente editável. A revisão final já acontece na plataforma; não há mais motivo para um ciclo de ida e volta por Word.

Esta ADR ficou despriorizada em 2026-07-16 (ver PROJECT_CONTEXT.md) porque nenhum capítulo tinha conteúdo revisado até o fim. Essa condição foi cumprida em 2026-07-20: o capítulo "Cura Emocional e Divino Feminino" (projeto "Isso vale ouro") chegou a `status: reviewed`, o primeiro do projeto inteiro a completar o pipeline até esse ponto.

## Decisões

### 1. Fonte de dados para exportação

A exportação lê sempre `chapters.manuscript_content` (o manuscrito já revisado), nunca recompõe a partir de `chapter_sources` diretamente — evita divergência entre o que foi de fato revisado e o que é exportado. Só capítulos com `status IN ('reviewed', 'final')` são exportáveis, mesmo gate humano já usado em todo o resto do pipeline.

### 2. Granularidade: por capítulo primeiro, livro inteiro depois

Fase A (esta ADR): exportação de **um capítulo por vez** — é o que já tem valor hoje, com 1 capítulo pronto. Endpoint novo `GET /chapters/<id>/export.docx`.

Fase B (futura, não coberta aqui): compilação do livro inteiro, concatenando todos os capítulos `reviewed`/`final` de um `book_project` na ordem de `chapter_order`. Fica para quando houver mais de um capítulo pronto — hoje seria um exagero de escopo para o valor real entregue.

### 3. Formato: só DOCX por enquanto

A ADR original já queria DOCX/PDF/EPUB desde o início. Esta revisão começa só por DOCX — é o mais simples de gerar a partir do manuscrito e o mais útil para revisão externa (abre em qualquer editor de texto). PDF e EPUB ficam para uma iteração futura, quando houver demanda real de leitura/distribuição, não de edição.

### 4. Geração técnica

Novo endpoint em `editorial-api` (Python/Flask, mesma stack de sempre): `GET /chapters/<id>/export.docx`.

- Verifica `status IN ('reviewed', 'final')`; recusa com 400 caso contrário.
- Percorre `manuscript_content` (documento Tiptap JSON) nó a nó, usando `python-docx`:
  - `literalSegment` → parágrafo com estilo visualmente distinto (recuo + itálico, com o título da canalização como legenda) — preserva a separação visual que já existe na tela do manuscrito (🔒).
  - `paragraph`/`text` com marks (`bold`/`italic`) → parágrafos comuns, marks traduzidas para *runs* formatados do Word.
- Cabeçalho do documento: título do capítulo, título do projeto.
- Resposta HTTP com `Content-Disposition: attachment` e o `.docx` como corpo binário.

### 5. Rastreabilidade

Mantém o princípio da ADR original, mas simplificado para o escopo por capítulo: `GET /chapters/<id>/export/manifest` (JSON separado, não embutido no `.docx`) lista `segment_id` → título/fonte de cada bloco literal do capítulo. Reconstituível a qualquer momento a partir do próprio `manuscript_content`, não precisa ser persistido à parte.

### 6. Fora de escopo nesta fase

- Promoção automática de `reviewed` para `final` ao exportar — a exportação pode ser repetida livremente enquanto o capítulo ainda está sendo lapidado; `final` continua sendo uma decisão humana separada, sem mecanismo de UI ainda (fica para quando o primeiro capítulo realmente for considerado pronto para publicação).
- PDF, EPUB, compilação do livro inteiro, distribuição/venda — como já dizia o desenho original, fora do objetivo declarado da plataforma (ADR-009).

## Consequências

- Fecha a primeira ponta do pipeline completo Vídeo → Transcrição → Segmento → Ficha → Mapa Filosófico → Projeto de Livro → Capítulo → **Exportação** definido na ADR-010, com rastreabilidade preservada do vídeo original até o parágrafo exportado — ainda que só no nível de um capítulo, não do livro inteiro.
- Nova dependência: `python-docx` em `editorial-api/requirements.txt`.
- Sem mudança de schema — reaproveita `manuscript_content` que a ADR-020 já criou.

## Nota de implementação (2026-07-20)

- `editorial-api`: `GET /chapters/<id>/export.docx` e `GET /chapters/<id>/export/manifest`, ambos atrás do gate `status IN ('reviewed', 'final')`. `build_chapter_docx()` percorre o Tiptap JSON com `python-docx` — `literalSegment` vira legenda em negrito+itálico ("Canalização — {título}") seguida de parágrafo recuado e itálico com o texto; `paragraph` vira parágrafo comum com `bold`/`italic` traduzidos para *runs*. Reaproveita `compose_manuscript_from_sources()` (já existente da ADR-020) como *fallback* para capítulos `reviewed` que nunca tiveram o manuscrito salvo explicitamente — foi o caso real do primeiro capítulo revisado do projeto, usado como teste de ponta a ponta.
- `editorial-ui`: `app/api/chapters/[chapterId]/export/route.ts` — proxy que injeta a `X-Service-Api-Key` no lado do servidor e transmite o `.docx` de volta (o navegador nunca vê a chave de serviço, mesmo padrão dos outros proxies já existentes). Link "Baixar DOCX" na tela do capítulo, visível só quando `status` é `reviewed`/`final`.
- Testado ponta a ponta contra dados reais de produção: gerado e validado o `.docx` do capítulo "Cura Emocional e Divino Feminino" (21 parágrafos, zip válido), confirmado o gate 400 num capítulo `draft`, e o proxy da `editorial-ui` testado sem browser via `curl` direto na rota Next.js.
- Deploy em produção: `editorial-api` revisão `editorial-api-00019-bgx`, `editorial-ui` revisão `editorial-ui-00019-bjz`.
- Fase B (compilação do livro inteiro) e PDF/EPUB permanecem fora de escopo, como já previsto na decisão — não implementados nesta passada.
