# ADR-019 - Insights Automáticos de Capítulos

**Status:** Proposta (2026-07-15)

## Contexto

Hoje a única forma de descobrir um capítulo candidato é o usuário abrir a `editorial-ui`, criar um projeto/capítulo e usar a busca ou o `/propose` manualmente (ADR-013/016). O usuário quer que a plataforma trabalhe proativamente: uma página de "insights" com cards de capítulos candidatos, gerados de forma assíncrona por um processo que varre o acervo periodicamente — clicando num card, o usuário vê o capítulo completo proposto e decide se promove para um projeto real ou descarta.

Isso é uma extensão natural do que já existe (`/propose` já monta capítulos algoritmicamente a partir de um escopo temático dado), mudando dois aspectos: (1) passa a ser a própria plataforma que decide *quais* escopos temáticos vale a pena propor, em vez de esperar o usuário escolher um; (2) passa a rodar em background, não sob demanda. O gate humano de sempre continua intacto — nada aqui é publicado ou vira capítulo de verdade sem uma ação explícita do usuário.

Ponto novo relevante: até hoje o uso de OpenAI no projeto é 100% embeddings (`text-embedding-3-small`, ADR-012). Gerar o título e o resumo de cada card exige uma chamada de geração de texto (chat completion) — a primeira do projeto. Isso é uma capacidade genuinamente nova, com uma linha de custo recorrente nova (pequena, mas real).

O usuário também pediu que a plataforma "aprenda" com sua forma de curar. Decisão já tomada explicitamente (ver ADR-017 quanto ao processo, e a pergunta direta feita ao usuário): esse aprendizado fica limitado a **estrutura e curadoria** — nunca gera prosa nova, nunca toca em conteúdo canalizado. Isso se encaixa como um sinal a mais dentro deste mesmo pipeline, não como um sistema separado.

## Decisão

### 1. Schema novo

`editorial.chapter_insights`:
- `id UUID PK`
- `title TEXT NOT NULL` — título candidato, gerado por IA
- `summary TEXT NOT NULL` — resumo curto para o card, gerado por IA
- `thematic_scope JSONB NOT NULL` — lista de `concept_id`s, mesmo formato de `chapters.thematic_scope`
- `proposed_sources JSONB NOT NULL` — snapshot completo e ordenado das fontes propostas (segmentos literais + fichas), mesmo formato de `chapter_revisions.sources_snapshot`
- `status TEXT NOT NULL DEFAULT 'suggested'` — CHECK em (`suggested`, `dismissed`, `promoted`)
- `promoted_chapter_id UUID NULL REFERENCES chapters(id)` — CHECK cruzado: `promoted_chapter_id IS NOT NULL` se e somente se `status = 'promoted'`
- `generated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()`
- `model TEXT NOT NULL` — identifica o modelo usado na geração, para rastreabilidade

### 2. Refatoração prévia obrigatória em `editorial-api/main.py`

`create_chapter` e `set_chapter_sources` hoje são funções de view Flask com lógica inline, cada uma abrindo sua própria conexão — não são chamáveis como funções internas reutilizáveis. Antes de implementar o endpoint de promoção, extrair a lógica de negócio de ambas para helpers puros `(cursor, ...)`, para que "promover um insight" rode como uma única transação atômica (criar capítulo + inserir sources + marcar insight como promovido), e não como duas chamadas HTTP encadeadas com risco de ficar pela metade em caso de falha.

### 3. Endpoints novos

- `POST /insights/generate` — **service-only** (chamado só pelo n8n agendado, nunca pelo browser/UI). Roda uma rodada de geração (ver §4).
- `GET /insights` — lista para os cards: `id, title, summary, thematic_scope, status, generated_at`.
- `GET /insights/<id>` — detalhe: capítulo proposto completo, com a mesma estrutura/renderização read-only que o `ChapterBuilder` já usa para mostrar fontes.
- `POST /insights/<id>/promote` — body `{book_project_id, chapter_order, title?}`. Transação única: cria o capítulo, insere as sources do snapshot, marca o insight como `promoted` com `promoted_chapter_id` preenchido.
- `POST /insights/<id>/dismiss` — marca como `dismissed`.

### 4. Lógica de geração

Reaproveita o ranking/dedup já usado em `POST /chapters/<id>/propose` (concept_relations + importance_score de ADR-012, dedup por similaridade de cosseno entre embeddings) para identificar clusters de conceitos com boa cobertura de conteúdo (segmentos/fichas relevantes) mas pouco representados nos capítulos já existentes de qualquer projeto. Para cada cluster candidato (lote limitado, ex. até 5 por rodada, para controlar custo):

1. Monta o `proposed_sources` do mesmo jeito que `/propose` já faz — blocos literais e fichas, nunca gerados ou reescritos.
2. Uma única chamada de chat completion da OpenAI gera `title` + `summary` curtos descrevendo o capítulo candidato — a IA nunca escreve o conteúdo do capítulo em si, só a etiqueta/resumo do card. Modelo recomendado: um modelo econômico da linha atual (ex. `gpt-4.1-mini` ou equivalente vigente no momento da implementação) — suficiente para um resumo curto, sem necessidade do modelo mais caro disponível.

### 5. Sinal de estilo (escopo aprovado: só estrutura, nunca prosa)

A mesma rotina de geração consulta os capítulos já `status = 'reviewed'` para calcular padrões observáveis: quais conceitos tendem a aparecer juntos no mesmo capítulo, tamanho/composição típica (proporção de blocos literais vs. fichas), convenções de escopo. Esses padrões só influenciam **quais clusters viram candidatos** e como são compostos — nunca geram texto novo. Calculado on-the-fly a cada rodada de geração, sem tabela de cache dedicada por enquanto (revisitar se o cálculo ficar caro com o crescimento do acervo de capítulos revisados).

### 6. Orquestração

Novo fluxo n8n agendado (ex. semanal, mesmo padrão dos Fluxos 01/02 já existentes), disparando `POST /insights/generate` com a chave de serviço (`X-Service-Api-Key`). Segue o princípio já estabelecido na ADR-009: n8n só orquestra/agenda, toda a lógica de negócio e acesso a dados fica na `editorial-api`.

### 7. UI

- `/insights` — grid de cards (título, resumo, chips de conceito, status).
- `/insights/<id>` — detalhe read-only do capítulo proposto (reaproveitando os componentes de renderização já existentes no `ChapterBuilder`/`SourceResultCard`), com ações "Promover para um projeto" (escolhe `book_project_id` + posição) e "Descartar".

## Consequências

- Primeira vez que o projeto usa geração de texto via LLM (não só embeddings) — introduz uma linha de custo recorrente nova, pequena e diretamente controlável pelo tamanho do lote e pela frequência do agendamento.
- O princípio "a IA organiza, nunca escreve como autora" (ADR-009/010) permanece intacto: a única prosa gerada por IA em todo este fluxo é o título/resumo do card, nunca o conteúdo do capítulo, que continua sendo exclusivamente blocos literais preservados e fichas já existentes.
- Cria o primeiro caso de "promoção" de um artefato gerado para um artefato real de projeto — o desenho da transação atômica de `/insights/<id>/promote` é o ponto mais delicado desta ADR e deve ser testado explicitamente contra falha parcial (ex. insight promovido mas sources não inseridas).
