# ADR-014 - Revisão Editorial e Consolidação de Capítulos

**Status:** Implementada (2026-07-15)

## Contexto

Uma vez que um projeto de livro (ADR-013) tenha múltiplos capítulos montados, dois problemas emergem que não existem quando se olha para um capítulo isolado: (1) o mesmo segmento ou trecho de destaque pode ter sido selecionado para mais de um capítulo, e (2) a obra como um todo precisa de coerência terminológica e de fluidez entre capítulos escritos em momentos diferentes. A ADR-009 define o Módulo 07 (Revisão) exatamente para isso.

## Decisões

### 1. Detecção de duplicidade entre capítulos

Usar os embeddings introduzidos na ADR-012 para detectar sobreposição de conteúdo entre `chapter_sources` de capítulos diferentes do mesmo `book_project`. A detecção apenas **sinaliza** a duplicidade (relatório de conflitos) — a remoção ou realocação é decisão humana, nunca automática, pelo mesmo princípio de autoria que rege a ADR-013.

### 2. Checklist de consolidação

Por capítulo, antes de avançar para `reviewed`:
- terminologia consistente com os nomes canônicos de `editorial.concepts` (ADR-011);
- transições escritas (`transition_context`) não alteram nem parafraseiam blocos `literal_segment` adjacentes;
- nenhuma canalização (`literal_segment`) aparece cortada no meio entre dois `chapter_sources`.

### 3. Registro de aprovação

`chapters.status` só avança para `reviewed` com `reviewed_by` e `reviewed_at` preenchidos — sempre um humano, nunca um processo automático.

## Consequências

A revisão prevista aqui é sobre organização, terminologia e fluidez — nunca sobre o conteúdo canalizado em si, que permanece imutável desde a preservação original (ADR-010). Isso fecha o ciclo de qualidade da obra sem violar o princípio de que a IA organiza e o autor é a fonte.

## Nota de implementação (2026-07-15)

- `GET /book-projects/<id>/duplicate-report` — usa os embeddings da ADR-012 num self-join de `chapter_sources` do mesmo projeto (`chapter_id` diferente), via `<=>` do pgvector. A mesma consulta cobre tanto reuso exato da mesma fonte (similaridade ≈ 1.0) quanto conteúdo apenas parecido — não precisou de uma query separada para cada caso. Só sinaliza, não corrige.
- `GET /chapters/<id>/consolidation-check` — três checagens determinísticas:
  1. mesma fonte usada mais de uma vez dentro do próprio capítulo;
  2. terminologia não-canônica: procura por *aliases* de conceitos (`editorial.concepts.aliases`) em texto de `transition_context` escrito manualmente, onde o nome canônico não aparece;
  3. possível paráfrase: embeda o texto de `transition_context` na hora e compara por similaridade de cosseno com os blocos `literal_segment` adjacentes (limiar 0.85, ajustável).
  
  A checagem "canalização cortada ao meio" **não foi implementada como scan** — é estruturalmente impossível dado o schema da ADR-013 (cada `chapter_source` referencia um segmento inteiro, nunca um trecho). O endpoint documenta isso na resposta em vez de fingir uma checagem que não faria sentido.
- `POST /chapters/<id>/review` — exige `reviewed_by` no corpo da requisição e `status = 'assembled'`; grava `reviewed_by`/`reviewed_at` e avança para `'reviewed'`. Não depende do resultado do checklist (a decisão continua sendo humana, o checklist é só informativo).

Testado de ponta a ponta em produção: 2 capítulos com o mesmo escopo temático propositalmente sobreposto → `duplicate-report` encontrou corretamente as 6 fontes reutilizadas (similaridade 1.000, mesmo segmento/ficha) → `consolidation-check` sem issues (esperado, capítulo sem texto de transição) → `review` registrou `reviewed_by` e avançou o status. Dados de teste removidos após validação.
