# ADR-016 - Curadoria Manual como Fluxo Principal (editorial-ui)

**Status:** Implementada (2026-07-15)

## Contexto

A ADR-013 introduziu `POST /chapters/<id>/propose`, um algoritmo determinístico que monta uma proposta inicial de fontes para um capítulo. No texto original da ADR-013, `/propose` é apresentado como o caminho natural de montagem, com `PUT /chapters/<id>/sources` citado como alternativa para "quando um humano quer curar a lista em vez de aceitar a proposta automática" — ou seja, a montagem manual era tratada como exceção.

Ao revisar o produto, ficou explícito que essa ênfase estava invertida: o usuário não quer que a plataforma "produza um livro" — quer navegar o acervo (segmentos, fichas, conceitos) e decidir manualmente, capítulo a capítulo, o que entra e o que não entra. Isso já era 100% suportado pelo backend (nenhuma mudança de schema ou endpoint foi necessária), mas faltava uma interface prática para fazer isso sem chamar a API na mão.

## Decisão

1. **Inversão de ênfase, não de capacidade.** `POST /chapters/<id>/propose` continua existindo e funcional, mas passa a ser tratado como atalho opcional — um ponto de partida que o usuário pode usar ou ignorar — nunca como o fluxo principal ou um passo obrigatório antes de `PUT /chapters/<id>/sources`. Nenhum endpoint foi removido ou alterado; a mudança é de que caminho a interface expõe como padrão.

2. **Novo serviço `editorial-ui`**, Next.js (App Router) no Cloud Run, replicando o padrão já estabelecido pelo `arquitetura-planner`: `--no-allow-unauthenticated`, acesso via `roles/run.invoker` restrito à conta do usuário, service account dedicada (`editorial-ui@thiago-ai-platform.iam.gserviceaccount.com`) com `secretmanager.secretAccessor` sobre o secret já existente `editorial-api-key` (reaproveitado, sem criar segredo novo).

3. **A UI não fala com Postgres nem com a OpenAI diretamente.** Toda leitura e escrita passa pela `editorial-api` já existente, usando a mesma credencial de serviço (`X-Service-Api-Key`), mantida só no servidor (Route Handlers e Server Actions do Next.js — nunca exposta ao browser).

4. **Telas:** lista/criação de projetos (`/`), detalhe do projeto com criação de capítulo por escopo temático (`/projects/[projectId]`), e o montador de capítulo (`/projects/[projectId]/chapters/[chapterId]`) — painel de exploração (busca semântica + filtro por conceito/tema/tipo) ao lado do painel de fontes já escolhidas (adicionar, remover, reordenar, salvar). Botões de status (`Aprovar`, `Marcar como revisado`) e o relatório de duplicidade (ADR-014) ficam acessíveis a partir daí.

## Consequências

Este é o primeiro serviço da plataforma cujo propósito é exclusivamente colocar um humano no meio do fluxo editorial de forma confortável — não adicionar automação. Otimizações futuras que tornem `/propose` mais sofisticado (ex. um modelo de ranking melhor) continuam válidas, mas não devem migrar a ênfase da interface de volta para "geração automática" sem uma decisão explícita em nova ADR, pelo mesmo princípio de autoria humana que rege a ADR-009 §10 e a ADR-013.

## Nota de implementação (2026-07-15)

Serviço `editorial-ui` implementado e implantado:

- Next.js 16 (App Router, Turbopack), TypeScript, Tailwind, `output: "standalone"`, Dockerfile multi-stage (`node:22-slim`).
- `lib/editorial-api.ts`: wrapper tipado server-only sobre todos os endpoints da `editorial-api` usados pela UI. As variáveis de ambiente (`EDITORIAL_API_URL`, `EDITORIAL_API_KEY`) são resolvidas dentro da função de request, não no escopo do módulo — isso evita que `next build` falhe ao coletar dados de página em ambientes (como o build do Cloud Build) onde essas variáveis só existem em runtime, não em tempo de build.
- Todas as páginas com dados dinâmicos (`/`, `/projects/[projectId]`, `/projects/[projectId]/chapters/[chapterId]`, `/projects/[projectId]/duplicate-report`) e a Route Handler `/api/concepts` foram marcadas explicitamente com `export const dynamic = "force-dynamic"` para impedir prerenderização estática em build — os dados do acervo mudam constantemente e nunca devem ser cacheados nesse nível.
- Testado ponta a ponta localmente (via Browser tool) contra a `editorial-api` de produção: criação de projeto, criação de capítulo com seleção de conceitos, busca semântica, navegação por conceito, adicionar/remover/reordenar fontes, salvar, `/propose`, aprovar (`draft` → `assembled`), revisar (`assembled` → `reviewed`), checklist de consolidação e relatório de duplicidade — sem erros de servidor. Dados de teste removidos após validação.
- Deploy em Cloud Run (`southamerica-east1`), revisão `editorial-ui-00001-mfr`, acesso IAM concedido a `contato@banhoseterapias.com`, verificado via `gcloud run services proxy editorial-ui`.
