# ADR-021 - Refresh de Design (shadcn/ui)

**Status:** Proposta (2026-07-15)

## Contexto

A `editorial-ui` (ADR-016) foi construída com Tailwind puro e componentes HTML simples — funcional, mas visualmente básica. O usuário pediu explicitamente uma UI "moderna e bonita" para a plataforma, principalmente porque as ADR-018 a 020 vão adicionar telas novas (login, insights, editor de manuscrito) que se beneficiam de um sistema de design mais consistente desde o início, em vez de nascerem no mesmo estilo básico e precisarem de retrabalho depois.

## Decisão

1. **Adotar shadcn/ui** sobre o stack já existente (Next.js + Tailwind) — não é uma biblioteca instalada como dependência, é um conjunto de componentes copiados para o repositório (Radix UI + Tailwind por baixo, licença MIT), o que significa zero vendor lock-in e controle total do código de cada componente.
2. **Tokens de design consistentes**: paleta de cores, tipografia e espaçamento centralizados (`tailwind.config`/CSS variables), em vez de classes Tailwind ad-hoc repetidas em cada tela como hoje.
3. **Dark mode** via `next-themes`, já que o restante das ferramentas do ecossistema (ex. artifacts) segue esse padrão de tema claro/escuro com `prefers-color-scheme` + override manual.
4. **Feedback consistente de ações**: toasts para sucesso/erro de Server Actions (criar, salvar, aprovar, revisar, promover) — hoje o feedback é implícito (a tela recarrega ou não muda), o que já causou confusão durante os testes da ADR-016.
5. **Navegação lateral persistente** (Projetos / Insights, a partir da ADR-019) substituindo o link único no cabeçalho atual.

## Não-decisão deliberada: sem rollout em bloco

Esta ADR não é seguida por uma fase de implementação própria e isolada. O rollout acontece **dentro** do trabalho de cada ADR seguinte (018, 019, 020): toda tela nova ou tela tocada por essas ADRs já nasce ou é migrada para os componentes do shadcn/ui, em vez de existir uma quinta fase de "redesign" depois que tudo o resto já estiver pronto. As telas hoje existentes (`/`, `/projects/[projectId]`, o montador de capítulo) são migradas oportunisticamente sempre que uma dessas ADRs precisar tocá-las — não há compromisso de migrar tudo de uma vez.

## Consequências

- Reduz o risco de inconsistência visual entre as telas antigas e as novas, sem exigir um projeto de redesign dedicado e isolado do resto do roadmap.
- Como não há uma "implementação da ADR-021" isolada, a nota de implementação final desta ADR só deve ser escrita depois que a ADR-020 (a última do roteiro atual) estiver concluída — neste ponto, dá para avaliar o que efetivamente foi migrado e o que ainda ficou no estilo antigo.
